# Brand Voice / 品牌意向

這份文件記錄我寫作跟與 AI 協作時的口吻偏好。Claude 在幫我寫文章、翻譯、調整文案時都會以這份為基準對照。

## 1. 核心定位

- **這個站**：個人內容平台，累積長期經驗。
- **文章區的目標讀者**：廣泛的科技愛好者，希望帶走「工具 + 方法」的組合。
- **個人 vs 品牌**：about 頁講「我自己」，品牌（LuMakes / 亮點研創）文案調整時不動個人能力描述，兩條敘事線分開維護。

## 2. 寫作口吻

- 第一人稱、口語化、像在跟朋友分享經驗
- **不用 emoji**
- 用「跟」不用「與」
- 語尾自然收：「搞定」「方便很多」「用起來很順」
- 不寫官方制式或商業文案語氣

## 3. 用詞偏好

- **避開過時口語**：「最殺」改用「最精妙」「最值得留下的」
- **帶選擇脈絡**：「因為我同時在用 A 跟 B，所以...」「試過 X 之後覺得 Y 更適合我」
- **第一人稱帶入**：「我自己」「我主要用」「我平常」
- **不要過度連接詞**：「然後」「就是」「其實」這類口語不要每段塞

## 4. 文章開場模式

- **開場列的痛點要對齊維度**：所有痛點要能被同一個共同根源解釋；不同維度的問題挪到後面當補充，不要混在開場
- **第一個 H2 必為底層邏輯**：先說明背後的方法或心智模型，再開始談工具
- **原則命名化**：把方法步驟取名（例如「先規劃、再動筆」），讀者好記
- **工具作證據**：不是工具列表，而是用工具來支撐前面提出的原則

## 5. 文章結構參考範本

寫新文章時可以對照這兩篇的 DNA：

- `src/content/articles/zh-Hant/file-management.mdx`
- `src/content/articles/zh-Hant/meeting-workflow.mdx`

它們的共通點是：開場先定錨原則 → 第一個 H2 講底層邏輯 → 後面的 H2 用工具佐證原則。

## 6. 工具標註格式：依場景挑強度

提到工具時的渲染樣式分兩種，依「這個工具是被當主角介紹，還是只是順帶提到」決定：

### 行內提及（inline）

段落中提到工具、bullet list 列舉工具的場景，用**簡單文字 + 括號標註費用**，不要塞 pricing-tag span，不要塞 official-link 大按鈕。連結用一般 Markdown `[文字](url)` 就好。

範例：

```mdx
- [Apple 備忘錄](https://example.com)：macOS 跟 iOS 內建，iCloud 自動同步（免費）
- [Bear](https://example.com)：介面乾淨、原生支援 Markdown（免費使用，進階功能需訂閱）
```

不要這樣：

```mdx
- [Bear](https://example.com)：介面乾淨 <span class="pricing-tag freemium">免費使用，進階功能需訂閱</span>
```

### 獨立卡片（標題 + 大圖 + 多段介紹）

工具推薦型文章的主要被介紹對象——每個工具獨立一個 H3，配大圖、多段 H4 講功能——這種場景才用 `pricing-tag` span 跟 `official-link` 大按鈕，放在介紹段落的最後當 CTA。

範例：

```mdx
### Bob

<Image src={bob} alt="Bob" />

[多段介紹...]

<span class="pricing-tag freemium">免費使用，付費買斷解鎖進階功能</span>
<a href="https://bobtranslate.com" target="_blank" class="official-link">Bob 官方網站</a>
```

### 判斷原則

- 只是順帶提到、bullet list 列工具 → **行內括號**
- 工具是這節的主角、有獨立 H3、有大圖、有多段 H4 → **pricing-tag + official-link 大按鈕**

兩種樣式不要混用——同一個工具不要既有 inline 括號又出現大按鈕。
