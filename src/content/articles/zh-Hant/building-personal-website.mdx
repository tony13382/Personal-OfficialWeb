個人網站這件事我前後搞了三次。第一次用 Bootstrap 從頭刻，第二次自己寫 Python 腳本生成 HTML，第三次才終於停在 Astro 上。每一次重寫都不是因為前一版壞了，而是因為我對「網站該怎麼管」的理解變了——從最早把它當成作品集，到後來當成內容平台，需求完全不一樣。

這篇重點不是技術選型清單，而是三次重寫之後定下來的一套流程跟心法，以及支撐它的框架跟平台選擇。先從三個版本一路怎麼演進講起。

**第一版：Bootstrap 一頁式靜態站**。當時只想要一個能放進履歷的連結，改內容靠複製貼上 HTML，換版型要動每張卡片——內容跟結構綁在一起，越疊越難維護。

**第二版：Python 腳本自己生 HTML**。把每個區塊包成 class 物件再 render 出來，開始摸到「資料跟渲染分離」的概念，但模板改起來超硬，加互動元件更麻煩。所有基礎建設（圖片壓縮、路由、SEO、RSS）都得自己寫，後來花在維護生成器的時間比寫內容還多。

**第三版：Astro + React，就是現在這個站**。這次的轉變不只是換工具，是把網站定位從「放作品的單頁名片」改成「持續累積內容的平台」。重點變成——**怎麼讓寫內容這件事變得夠輕**。

## 框架跟部署都選在 Astro + Cloudflare

### 為什麼選 Astro

選 Astro 主要是因為它剛好踩在「內容驅動」跟「元件化」的交會點——內容用 MDX 寫，靜態部分編譯成純 HTML，真正需要互動的地方再用 React 元件包成 island 嵌進去。

挑框架時我的四個硬需求是：

- 能輕鬆部署到靜態平台（尤其 Cloudflare，因為我網域跟 DNS 都在那）
- 不需要 SSR、不需要 API server
- 能混合 React 互動元件
- 內容能用 MDX 寫

比較過 Next.js（功能太全、重心在 SSR、不想綁 Vercel）、Hugo（不支援 MDX，自訂元件要用 Shortcodes）、Nuxt（我 Vue 不熟），只有 Astro 四點全中。

用下來真正最有感的是 **Content Collections**。每個 collection 都有 Zod schema，寫 MDX 時欄位漏填或格式錯，build 前 IDE 就紅線提醒。以 articles collection 為例：

<p class="code-title">src/content/config.ts</p>

```ts
const articlesCollection = defineCollection({
  type: 'content',
  schema: z.object({
    title: z.string(),
    description: z.string(),
    date: z.coerce.date(),
    category: z.string(),
    tags: z.array(z.string()).default([]),
    pin: z.boolean().default(false),
    cover: z.string().optional(),
  })
})
```

`date` 打成 `2024-13-40` 這種非法日期、或漏填 `category`，`astro dev` 當場就噴錯。以前第二版的 Python 生成器要等跑完才知道欄位錯了，現在寫的時候就擋下來。

加上 MDX 本身就是「可互動的內容容器」——單篇可以 import React 元件，跨文章可以用 `getCollection('articles')` 把所有 frontmatter 當陣列操作，置頂、分類、標籤全部靠欄位驅動。發新文章的成本被壓到只剩「寫一個 MDX 檔」，列表頁、標籤頁會自動更新。

### 部署也走 Cloudflare

前兩版都跑在 GitHub Pages。兩邊對個人站都很夠用，我選遷過去主要是這幾點：

- **Workers 生態**（D1、KV、R2、Durable Objects）讓未來要加動態功能時資源都現成
- **網域本來就在 Cloudflare**，Pages 綁網域同一個儀表板點一下，HTTPS 自動發
- 邊緣節點多，台灣延遲比 Vercel、Netlify 更順

遷移時的第一次設定大概是三步：

**1. 連 Git 來源**：選「Continue with GitHub」授權 Cloudflare 讀 repo 列表。

<Image src={cloudflareSetup1} alt="Cloudflare Workers & Pages 建立頁面，選 Continue with GitHub" />

**2. 挑 repo**：下拉選擇帳號之後挑網站對應的 repo，按下一步。

<Image src={cloudflareSetup2} alt="Cloudflare 選擇 GitHub repository 的畫面" />

**3. 填建置指令**：組建命令輸入 `npm run build`，部署命令填 `npx wrangler deploy`（走 Workers 統一流程），按「部署」就會跑第一次 build。

<Image src={cloudflareSetup3} alt="Cloudflare 建立應用程式時設定組建跟部署命令的畫面" />

設定好之後，後台的建置頁面長這樣——之後要調 Node 版本、輸出目錄、分支名稱都在這邊：

- **Framework preset**：Astro（選下去 build command 跟 output 會自動帶好）
- **Build command**：`npm run build`
- **Build output directory**：`dist`
- **Node version**：用 `NODE_VERSION=20` 環境變數鎖版本

<Image src={cloudflareBuildSettings} alt="Cloudflare Pages 後台的建置設定畫面" />

綁完之後推 main 分支自動 build、自動部署到 edge，整個流程 1–2 分鐘。自訂網域點一下「設定自訂網域」輸入名稱就好，CNAME 跟 HTTPS 憑證自動處理。

<Image src={cloudflareCustomDomain} alt="Cloudflare Pages 自訂網域設定畫面" />

### 網域也在同一個平台

網域我走 Cloudflare Registrar 註冊，理由是「成本價」模式——多少錢就收多少錢，不會有第一年便宜、續費翻倍的陷阱，WHOIS 隱私也免費內建。其他註冊商（GoDaddy、Namecheap）比較下來都有續費漲價、介面雜亂、加購推銷這些扣分項。

命名幾個原則：好念好拼、不要塞職業或領域（`xxx-designer.com` 換跑道會尷尬）、TLD 選單純的（`.com` / `.me` / `.dev`）、長度壓短避免連字號。實驗性的 side project 走子網域（`app.lumakes.com`、`lab.lumakes.com`）——**主站是身份，子站是實驗**，維護一個品牌樹就夠。

## 現在的管理流程：Claude Code 管樣式，MDX 寫內容

### 職責分離

到了這一版，我慢慢摸出一個對自己最順的工作分工——**樣式跟結構用 Claude Code 管、內容用 MDX 自己寫**。

| 類型 | 工具 | 為什麼這樣分 |
|---|---|---|
| 版型、元件、樣式 | Claude Code + Tailwind | 技術決策，需要對整個專案有認知，Claude 效率更高 |
| 文章、專案、經歷的文字 | 自己用 MDX 寫 | 創作，需要個人視角跟語氣，AI 沒辦法代勞 |

把這兩件事分開之後，我寫文章的時候不用被樣式打斷，調樣式的時候也不用切換到「寫作模式」。

### 樣式這條線：交給 Claude Code

版型、元件、CSS 這類工作是 Claude Code 最擅長的地方。它能一次讀完整個專案的上下文，理解我用的是 Tailwind v4 不是 v3、shadcn/ui 的元件放在哪、主題變數怎麼定，改起來的一致性比我自己動手還穩。

關鍵是要寫好 `CLAUDE.md`。我在專案根目錄放了一份，涵蓋技術棧、資料夾結構、樣式規則、常見地雷。每次開新 session，Claude 一啟動就知道這個專案在幹嘛，不用重複解釋。之前寫過一篇 [Claude Code 調教筆記](/articles/claude-code-setup)，細節都在那邊。

### 內容這條線：從靈感到發佈

寫文章的流程我切成四步：靈感蒐集 → 先規劃 → 再動筆 → Skill 化。

**靈感蒐集：求快不求精**

第一步是靈感蒐集，這個階段我講求的是**快**，不是精準——有想法就要馬上記下來，格式亂、錯字多都無所謂，重點是不要讓思緒流走。我挑的工具都是那種**打開就能寫**的輕量速記軟體，不會硬逼我分類或下 tag：

- [Apple 備忘錄](https://support.apple.com/zh-tw/guide/notes/welcome/mac)：macOS 跟 iOS 內建，iCloud 自動同步，完全不用設定 <span class="pricing-tag">免費</span>
- [Bear](https://bear.app/)：介面乾淨、原生支援 Markdown，從短訊速記到長文大綱都能接 <span class="pricing-tag freemium">免費使用，進階功能需訂閱</span>
- [SimpleNote](https://simplenote.com/)：超輕量的純文字筆記，開啟速度很快，跨平台同步 <span class="pricing-tag">免費</span>

這三個共通點都一樣——**無摩擦**。靈感蒐集最怕的就是「整理感」太重，一旦要決定「這筆記該放哪個資料夾」就會卡住，想法也就跟著冷掉。

**先規劃、再動筆**

每次要寫新文章，我會先跟 Claude 討論三件事——主題、slug、H2 結構。這三個確定之後，整篇的骨架就定了，剩下就是填肉。討論這一步其實很重要，它幫我把模糊的想法逼成具體的段落劃分，避免動筆之後才發現結構不對要整篇重組。

**用 Skill 包起來**

我把上面的流程包成一個叫 `new-article` 的自訂 Skill。指令裡面預先寫好文章的三種類型（工具推薦型、方法論型、經驗分享型）、每種類型該怎麼結構化、可用的特殊元件（pricing tag、callout、code title）長什麼樣、口吻規範。之後我只要打 `/new-article`，Claude 就會進入「文章撰寫模式」，按照這套 SOP 走一遍。

Skill 裡我覺得最關鍵的是「Writing Style Guide」這一節，直接把我希望 Claude 維持的口吻規則寫死：

<p class="code-title">~/.claude/skills/new-article/skill.md</p>

```md
## Writing Style Guide

- 口語化、像在跟朋友分享自己的經驗
- 第一人稱帶入：「我自己」「我主要用」「我平常」
- 交代選擇脈絡：「因為我同時在用 A 跟 B，所以...」
- 用「跟」不用「與」
- 不要用 emoji
- 語尾自然收：「搞定」「方便很多」「用起來很順」
```

另外還有一張文章類型對照表，定義「工具推薦型 / 方法論型 / 經驗分享型」各自的重點，Claude 開工前會先問我這篇是哪一種，再挑對應的骨架走。這些規則如果每次開新對話都要重講一遍太耗神，包進 Skill 之後 `/new-article` 一打進去整套規則就自動 loaded。

把這套流程 Skill 化之後，**我每次的產出格式都一致、口吻都穩定**，不用每篇文章都重新跟 Claude 溝通規則。整個站的文章區風格能維持一致，靠的就是這個 Skill。

**MDX 直接寫，不走 CMS**

我沒有接任何 headless CMS。文章就是一個個 `.mdx` 檔案，放在 `src/content/articles/` 底下，用 Git 管版本。好處是很單純：

- 寫作跟程式碼在同一個編輯器、同一個 Git 工作流裡面
- 不用維護額外的後台服務，也不用擔心 CMS 供應商倒掉或漲價
- 文章內容是純文字，未來要遷移到任何平台都很容易

對只有我一個人寫的個人站來說，MDX 比任何 CMS 都輕，而且本身就能混進 React 元件、Astro 的 `<Image>`，需要的表達力完全夠。

### 發佈節奏：讓 Git 推著跑

流程的最後一環是發佈。我的做法就是——**Cloudflare Pages 盯著 main 分支，推上去就自動 build 跟上線**。沒有 staging、沒有 preview flow，因為這就是我自己的站，有錯就改完再推一次，不需要複雜的部署流程。

## 三次重寫學到的心法

三次重寫下來，我把這些經驗濃縮成幾條會推給想做個人站朋友的心法：

**一、定位決定一切，不是工具**。第一版當履歷、第二版當作品集、第三版當內容平台——每次「重建」不是因為舊工具壞了，是因為定位變了。先想清楚這個站對自己是什麼，再選工具。

**二、內容跟結構要能分離**。第一版 HTML 手刻、第二版把樣式寫死在 Python class 裡，兩次都是敗在「換版型要動每一篇」。現在內容走 MDX、樣式走 Tailwind + 元件，兩邊可以各自演進。

**三、別自己重造基礎建設**。第二版我花了一堆時間寫圖片處理、路由、SEO——這些 Astro 都內建了。自造工具看起來自由，其實每一樣都在消耗寫內容的力氣。

**四、把重複的事 Skill 化**。發文章的節奏之所以能穩定，不是靠紀律，是靠 `new-article` Skill 把規則固化下來。想讓一件事持續做，要先降低它每次開始的摩擦。

**五、小站就該有小站的節奏**。沒有 staging、沒有 preview flow、push main 就上線。過度工程化反而會讓自己不想寫東西——這個站是為了累積內容存在，不是為了展示架構能力。

這一版我大概能用很久了——因為卡住的部分不是技術，是我對「網站該怎麼存在」的理解。
